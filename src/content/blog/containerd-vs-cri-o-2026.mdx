This is a practical comparison based on real production use, not vendor marketing.

## Quick Comparison

| Feature | containerd | CRI-O |
|---------|-----------|-------|
| Scope | General-purpose container runtime | CRI-only (Kubernetes-focused) |
| Used by | Docker, K3s, EKS, GKE, AKS | OpenShift, Kubernetes |
| Image building | Via BuildKit/nerdctl | Not included (use Buildah) |
| CLI tools | ctr, nerdctl, crictl | crictl |
| Memory footprint | Slightly larger | Minimal |
| OCI compliance | Full | Full |
| Plugins | Snapshotter plugins | None needed |

## When to Use containerd

- **Default choice**: containerd is the runtime for Docker, K3s, EKS, GKE, and AKS
- **Multi-purpose**: Works beyond Kubernetes — standalone containers, CI/CD, development
- **Docker compatibility**: nerdctl provides Docker-compatible CLI on top of containerd
- **Ecosystem**: Largest community and widest adoption

## When to Use CRI-O

- **OpenShift**: CRI-O is the default and only supported runtime for Red Hat OpenShift
- **Minimal footprint**: CRI-O implements only the CRI specification — nothing more
- **Kubernetes-only**: If your runtime only needs to serve Kubernetes, CRI-O is purpose-built
- **Version-aligned releases**: CRI-O versions match Kubernetes versions (1.30, 1.31, etc.)

## The Practical Truth

For most teams, the choice is already made:
- **Using a managed service?** containerd (EKS, GKE, AKS all default to containerd)
- **Using OpenShift?** CRI-O (no choice)
- **Building your own cluster?** Either works — containerd has more tooling, CRI-O has a smaller surface

## My Recommendation

Use **containerd** for general-purpose clusters and managed Kubernetes. Use **CRI-O** for OpenShift or when you want the absolute minimum runtime surface area.
