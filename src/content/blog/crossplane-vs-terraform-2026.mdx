This is a practical comparison based on real production use, not vendor marketing.

## Quick Comparison

| Feature | Crossplane | Terraform |
|---------|-----------|-----------|
| Model | Kubernetes-native (CRDs) | CLI + state file |
| Reconciliation | Continuous (control loop) | On-demand (`terraform apply`) |
| Language | YAML (Kubernetes manifests) | HCL |
| State | Kubernetes etcd | Terraform state file |
| Drift detection | Automatic (continuous) | Manual (`terraform plan`) |
| Composability | Compositions (XRDs) | Modules |
| GitOps | Native (ArgoCD/Flux) | Requires wrapper |
| Multi-tenancy | Kubernetes RBAC | Terraform Cloud/Enterprise |

## When to Use Crossplane

- **Kubernetes-native teams**: If your team already thinks in Kubernetes resources, Crossplane feels natural
- **Continuous reconciliation**: Crossplane detects and fixes drift automatically — infrastructure self-heals
- **GitOps integration**: Manage infrastructure through ArgoCD or Flux, same as application deployments
- **Platform engineering**: Build custom APIs (Compositions) that abstract cloud complexity for developers

```yaml
# Crossplane: Developer requests a database
apiVersion: database.example.com/v1alpha1
kind: PostgreSQLInstance
metadata:
  name: my-db
spec:
  parameters:
    size: small
    region: eu-west-1
```

## When to Use Terraform

- **Multi-cloud IaC standard**: Terraform is the most widely adopted IaC tool with the largest provider ecosystem
- **Mature tooling**: `terraform plan` preview, import existing resources, extensive documentation
- **Team familiarity**: Most infrastructure engineers already know HCL
- **Terraform Cloud**: Managed state, policy-as-code (Sentinel), run approvals, cost estimation

```hcl
# Terraform: Explicit resource definition
resource "aws_db_instance" "my_db" {
  engine         = "postgres"
  instance_class = "db.t3.medium"
  allocated_storage = 20
}
```

## The Convergence

In 2026, the best teams use both:
- **Terraform** for initial infrastructure provisioning and one-time setups
- **Crossplane** for ongoing infrastructure management with continuous reconciliation

## My Recommendation

Use **Terraform** if your team is infrastructure-focused and you need the broadest provider support. Use **Crossplane** if you are building a platform engineering layer and want infrastructure management to follow Kubernetes patterns.
