This is a practical comparison based on real production use, not vendor marketing.

## Quick Comparison

| Feature | Jaeger | Zipkin |
|---------|--------|--------|
| Origin | Uber | Twitter |
| CNCF status | Graduated | Not CNCF |
| Backend storage | Cassandra, Elasticsearch, Kafka, Badger | Cassandra, Elasticsearch, MySQL |
| UI | Modern, feature-rich | Simple, functional |
| Sampling | Adaptive (head + tail) | Head-based only |
| OpenTelemetry | Native support | Via collector |
| Deployment | Kubernetes operator | Docker/manual |
| Analytics | Service dependencies, DAG | Service dependencies |

## When to Use Jaeger

- **Kubernetes environments**: Jaeger has a Kubernetes Operator for easy deployment and management
- **OpenTelemetry**: Jaeger natively supports OTLP protocol — the industry standard
- **Adaptive sampling**: Jaeger can adjust sampling rates per-service based on traffic volume
- **CNCF ecosystem**: Integrates naturally with Prometheus, Grafana, and Kubernetes

## When to Use Zipkin

- **Simplicity**: Zipkin is easier to set up and operate — single binary, minimal configuration
- **Existing Zipkin instrumentation**: If your services already use Zipkin client libraries
- **Lightweight requirements**: When you need basic distributed tracing without the full observability platform

## The 2026 Reality

**OpenTelemetry has become the standard**. Both Jaeger and Zipkin can receive OpenTelemetry data, but the trend is:

1. Instrument with **OpenTelemetry SDK**
2. Collect with **OpenTelemetry Collector**
3. Store in **Jaeger**, **Tempo**, or a commercial backend

**Grafana Tempo** is increasingly popular as a third option — it uses object storage (like Loki for logs) making it significantly cheaper to operate than either Jaeger or Zipkin with Elasticsearch.

## My Recommendation

Use **Jaeger** for production Kubernetes environments, especially in the CNCF ecosystem. Consider **Grafana Tempo** if you already use the Grafana stack (Prometheus + Loki + Grafana).
