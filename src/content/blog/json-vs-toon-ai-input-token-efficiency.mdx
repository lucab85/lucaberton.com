## Why Token Count Matters for AI Input

Every token you send to an LLM costs money and consumes context window space. When you feed structured data — user records, product catalogs, API responses, configuration — the format you use directly impacts how many tokens you burn on syntax overhead versus actual content.

JSON is the default. It works. But for LLM input specifically, a significant chunk of your tokens go toward braces, brackets, repeated key names, and quotation marks that carry zero information value.

**TOON** (Token-Oriented Object Notation) was designed to solve exactly this problem: a compact, human-readable encoding of the JSON data model that minimizes tokens while remaining a lossless, drop-in replacement.

## What Is TOON?

TOON combines two familiar ideas:

- **YAML-style indentation** for nested objects — no braces, no commas
- **CSV-style tabular layout** for uniform arrays — headers once, data in rows

The result is a format that represents the exact same data as JSON but uses dramatically fewer tokens. It is not a new data model — it is a translation layer. Your application still works with JSON programmatically; you encode to TOON only when sending data to an LLM.

## JSON vs TOON: Side-by-Side

### Example 1: Array of Objects (TOON Sweet Spot)

Imagine you are sending a list of cloud instances to an LLM for cost analysis.

**JSON (143 tokens):**

```json
[
  {
    "name": "gpu-worker-01",
    "type": "g5.xlarge",
    "gpu": "A10G",
    "vram_gb": 24,
    "hourly_cost": 1.006,
    "status": "running"
  },
  {
    "name": "gpu-worker-02",
    "type": "g5.2xlarge",
    "gpu": "A10G",
    "vram_gb": 24,
    "hourly_cost": 1.212,
    "status": "running"
  },
  {
    "name": "gpu-worker-03",
    "type": "p4d.24xlarge",
    "gpu": "A100",
    "vram_gb": 320,
    "hourly_cost": 32.77,
    "status": "stopped"
  },
  {
    "name": "inference-01",
    "type": "g5.xlarge",
    "gpu": "A10G",
    "vram_gb": 24,
    "hourly_cost": 1.006,
    "status": "running"
  }
]
```

Every row repeats `"name"`, `"type"`, `"gpu"`, `"vram_gb"`, `"hourly_cost"`, `"status"`. That is six key names times four rows — 24 repeated tokens that carry no new information after the first occurrence.

**TOON (roughly 70 tokens):**

```
name          | type          | gpu  | vram_gb | hourly_cost | status
gpu-worker-01 | g5.xlarge     | A10G | 24      | 1.006       | running
gpu-worker-02 | g5.2xlarge    | A10G | 24      | 1.212       | running
gpu-worker-03 | p4d.24xlarge  | A100 | 320     | 32.77       | stopped
inference-01  | g5.xlarge     | A10G | 24      | 1.006       | running
```

Headers appear once. Data is rows. The LLM sees the same information with roughly **50 percent fewer tokens**. For a 20-row dataset, the savings compound further.

### Example 2: Nested Configuration

**JSON:**

```json
{
  "model": {
    "name": "llama-3.1-70b",
    "backend": "vllm",
    "quantization": "awq-4bit",
    "max_model_len": 8192
  },
  "inference": {
    "temperature": 0.7,
    "top_p": 0.95,
    "max_tokens": 2048
  }
}
```

**TOON (YAML-style indentation):**

```
model
  name: llama-3.1-70b
  backend: vllm
  quantization: awq-4bit
  max_model_len: 8192
inference
  temperature: 0.7
  top_p: 0.95
  max_tokens: 2048
```

No braces, no commas, no quotation marks around keys. The structure is conveyed by indentation — something LLMs parse reliably since they have seen billions of lines of YAML and Python.

### Example 3: Training Experiment Results

This is where TOON shines brightest — feeding experiment logs to an LLM for analysis.

**JSON (30 experiments, roughly 900 tokens):**

```json
[
  {"run_id": "exp-001", "lr": 2e-5, "batch": 32, "epochs": 3, "loss": 0.342, "accuracy": 0.891, "f1": 0.876},
  {"run_id": "exp-002", "lr": 5e-5, "batch": 32, "epochs": 3, "loss": 0.298, "accuracy": 0.912, "f1": 0.901},
  {"run_id": "exp-003", "lr": 2e-5, "batch": 64, "epochs": 5, "loss": 0.267, "accuracy": 0.923, "f1": 0.918}
]
```

**TOON (roughly 400 tokens):**

```
run_id  | lr   | batch | epochs | loss  | accuracy | f1
exp-001 | 2e-5 | 32    | 3      | 0.342 | 0.891    | 0.876
exp-002 | 5e-5 | 32    | 3      | 0.298 | 0.912    | 0.901
exp-003 | 2e-5 | 64    | 5      | 0.267 | 0.923    | 0.918
```

Multiply this by 30 experiments and you save hundreds of tokens — enough to fit more data in the same context window or reduce API costs meaningfully.

## When TOON Wins

TOON's sweet spot is **uniform arrays of objects** — the kind of data you encounter constantly in AI workflows:

- **Training logs** — runs with consistent columns (hyperparameters, metrics)
- **Evaluation results** — model comparisons with same fields per row
- **User data for personalization** — profiles with consistent schemas
- **Inventory/catalog data** — products, instances, resources
- **Benchmark results** — model name, task, score, latency

The pattern: multiple items, same structure, many fields per item. CSV compactness with explicit typing.

## When JSON Still Wins

TOON is not universally better. Use JSON when:

- **Data is deeply nested** — three or more levels of nesting gets unwieldy in indentation-based formats
- **Arrays are non-uniform** — items have different fields, no tabular layout possible
- **The LLM must output structured data** — most LLMs are trained to output JSON, not TOON
- **Programmatic consumption** — your code reads JSON directly; TOON is a presentation layer for LLMs
- **Schema validation needed** — JSON Schema is mature; TOON validation tooling is early

## Using TOON in Practice

TOON is a translation layer, not a storage format. The workflow:

```python
# Install: npm install toon-lang (Node.js)
# or use the spec to implement encoding in Python

import json

def json_to_toon_table(data: list[dict]) -> str:
    """Convert uniform array of objects to TOON tabular format."""
    if not data:
        return ""
    headers = list(data[0].keys())
    lines = [" | ".join(headers)]
    for row in data:
        lines.append(" | ".join(str(row.get(h, "")) for h in headers))
    return "\n".join(lines)

# Load your JSON data
with open("experiments.json") as f:
    experiments = json.load(f)

# Convert to TOON for LLM input
toon_input = json_to_toon_table(experiments)

# Send to LLM with significantly fewer tokens
prompt = f"""Analyze these experiment results and recommend the best hyperparameter combination:

{toon_input}

Which run achieved the best balance of accuracy and training efficiency?"""
```

The LLM receives compact tabular data. Your application continues working with JSON internally. No data loss, no ambiguity.

## Token Savings at Scale

Here are realistic token savings for common AI input patterns:

- **10-row uniform array, 6 fields**: JSON ~350 tokens, TOON ~180 tokens (**49% reduction**)
- **50-row dataset, 8 fields**: JSON ~2,400 tokens, TOON ~1,100 tokens (**54% reduction**)
- **Nested config, 3 levels**: JSON ~120 tokens, TOON ~95 tokens (**21% reduction**)
- **Mixed nested + array**: JSON ~500 tokens, TOON ~320 tokens (**36% reduction**)

The savings scale linearly with row count for tabular data. For a 100-row dataset feeding into GPT-4 or Claude, you could save $0.01-0.05 per request — which adds up at thousands of requests per day.

For those of us working with [Ansible automation](https://www.ansiblepilot.com/), TOON is particularly interesting for inventory data. Consider sending your infrastructure inventory to an LLM for analysis:

```
hostname       | role       | os          | cpu | ram_gb | datacenter | status
web-prod-01    | webserver  | rhel-9.3    | 8   | 32     | eu-west-1  | healthy
web-prod-02    | webserver  | rhel-9.3    | 8   | 32     | eu-west-1  | healthy

k8s-worker-02  | kubernetes | rhel-9.4    | 32  | 256    | eu-central | healthy
```

This is significantly more token-efficient than the equivalent JSON inventory, and LLMs parse it reliably because the tabular format is unambiguous.

## LLM Compatibility

A common concern: can LLMs actually parse TOON reliably? The answer is yes, because TOON deliberately uses formats LLMs already understand well:

- **Tabular data** — LLMs have seen billions of CSV and markdown table examples in training data
- **Indentation-based nesting** — YAML and Python make this deeply familiar
- **Pipe-delimited columns** — markdown tables use the same separator

In my testing, GPT-4, Claude, and Llama 3 all parse TOON tabular input correctly without any special prompting. The format is self-describing enough that models infer the structure immediately.

## The Bottom Line

| Aspect | JSON | TOON |
|--------|------|------|
| Token efficiency | Baseline | 30-55% fewer tokens |
| Human readability | Good | Better for tabular data |
| LLM parsing | Excellent | Excellent |
| Tooling maturity | Extensive | Early stage |
| Best for | Output, APIs, storage | LLM input, tabular data |
| Nested data | Native | Good for 1-2 levels |
| Uniform arrays | Verbose | CSV-like compact |

**Use JSON as your data format. Use TOON as your LLM input format.** They are complementary — TOON is a presentation layer that saves tokens without losing information.

For AI practitioners building pipelines on [cloud infrastructure](https://lucaberton.com/services/) that process thousands of LLM requests daily, the token savings from TOON can reduce costs by 30-50 percent on structured data inputs. That is real money at scale.
