This is a practical comparison based on real production use, not vendor marketing.

## Quick Comparison

| Feature | KEDA | HPA (Horizontal Pod Autoscaler) |
|---------|------|------|
| Metrics sources | 60+ scalers (Kafka, SQS, Prometheus, etc.) | CPU, memory, custom metrics |
| Scale to zero | Yes | No (minimum 1 replica) |
| Event-driven | Yes (queue length, cron, external events) | No (metric threshold only) |
| Built-in | No (install separately) | Yes (Kubernetes core) |
| Complexity | Medium | Low |
| CNCF status | Graduated | Core Kubernetes |

## When to Use HPA

- **Simple scaling**: CPU and memory-based scaling is sufficient for most web applications
- **No additional dependencies**: HPA is built into every Kubernetes cluster
- **Predictable workloads**: Request-based services with gradual traffic changes

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

## When to Use KEDA

- **Queue-based workloads**: Scale workers based on Kafka lag, RabbitMQ queue depth, or AWS SQS messages
- **Scale to zero**: Save costs by scaling idle workloads to zero replicas
- **External metrics**: Scale based on Prometheus queries, Datadog metrics, or any custom source
- **Cron-based scaling**: Pre-scale before expected traffic spikes
- **Event-driven architectures**: Serverless-like scaling on Kubernetes

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: kafka-consumer
spec:
  scaleTargetRef:
    name: kafka-consumer
  minReplicaCount: 0          # Scale to zero!
  maxReplicaCount: 50
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: kafka:9092
        consumerGroup: my-group
        topic: events
        lagThreshold: "100"
```

## KEDA Works With HPA

KEDA does not replace HPA — it creates HPA resources under the hood and extends them with external metrics. You can use both: KEDA for event-driven scaling and HPA for CPU/memory.
