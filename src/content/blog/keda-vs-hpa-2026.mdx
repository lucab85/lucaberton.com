HPA (Horizontal Pod Autoscaler) is built into Kubernetes and scales based on CPU/memory. KEDA extends HPA with 60+ event sources and scale-to-zero. KEDA does not replace HPA — it enhances it.

## How they work

### HPA

HPA watches metrics and adjusts replica count:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
```

HPA checks metrics every 15 seconds and adjusts replicas. Simple, reliable, built-in.

### KEDA

KEDA creates HPA objects behind the scenes but adds event-driven triggers:

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: worker-scaler
spec:
  scaleTargetRef:
    name: worker
  minReplicaCount: 0    # Scale to zero!
  maxReplicaCount: 50
  triggers:
    - type: rabbitmq
      metadata:
        host: amqp://rabbitmq:5672
        queueName: tasks
        queueLength: "10"    # 1 pod per 10 messages

    - type: prometheus
      metadata:
        serverAddress: http://prometheus:9090
        query: sum(rate(http_requests_total[2m]))
        threshold: "100"
```

KEDA watches external event sources (message queues, databases, custom metrics) and creates/manages HPA objects automatically.

## Feature comparison

| Feature | HPA | KEDA |
|---|---|---|
| **Built-in** | Yes (Kubernetes core) | Addon (install separately) |
| **Metric sources** | CPU, memory, custom metrics API | 60+ scalers (Kafka, RabbitMQ, Prometheus, AWS SQS, etc.) |
| **Scale to zero** | No (min 1 replica) | Yes |
| **Scale from zero** | N/A | Yes (activates on events) |
| **External metrics** | Requires custom metrics adapter | Built-in for each scaler |
| **Cron scaling** | No | Yes (scheduled scaling) |
| **Multiple triggers** | Multiple metrics (AND logic) | Multiple triggers (OR logic) |
| **Cooldown** | stabilizationWindow | cooldownPeriod |
| **Jobs** | No | Yes (ScaledJob for batch work) |

## Common KEDA scalers

```yaml
# Kafka consumer lag
- type: kafka
  metadata:
    bootstrapServers: kafka:9092
    consumerGroup: my-group
    topic: events
    lagThreshold: "50"

# AWS SQS queue depth
- type: aws-sqs-queue
  metadata:
    queueURL: https://sqs.us-east-1.amazonaws.com/123/my-queue
    queueLength: "5"
    awsRegion: us-east-1

# PostgreSQL query result
- type: postgresql
  metadata:
    connectionFromEnv: PG_CONNECTION
    query: "SELECT count(*) FROM jobs WHERE status = 'pending'"
    targetQueryValue: "10"

# Cron schedule
- type: cron
  metadata:
    timezone: Europe/Amsterdam
    start: "0 8 * * 1-5"   # Scale up weekdays 8am
    end: "0 20 * * 1-5"    # Scale down weekdays 8pm
    desiredReplicas: "10"

# HTTP request rate (KEDA HTTP Add-on)
- type: prometheus
  metadata:
    query: sum(rate(nginx_ingress_controller_requests[2m]))
    threshold: "500"
```

## Scale-to-zero (KEDA advantage)

KEDA's scale-to-zero saves costs for bursty workloads:

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: inference-scaler
spec:
  scaleTargetRef:
    name: llm-inference
  minReplicaCount: 0          # Zero pods when idle
  maxReplicaCount: 10
  pollingInterval: 10
  cooldownPeriod: 300         # Wait 5 min before scaling to zero
  triggers:
    - type: prometheus
      metadata:
        query: sum(kube_deployment_status_replicas_available{deployment="llm-inference"})
        threshold: "1"
```

With GPU workloads at $30+/hour per node, scaling to zero during idle periods saves thousands per month.

## Decision guide

### Choose HPA when:
- **CPU/memory autoscaling** is sufficient
- You want **zero dependencies** — HPA is built into every Kubernetes cluster
- Simple web services that scale with **request load**
- You already use **custom metrics API** (Prometheus adapter)

### Choose KEDA when:
- **Event-driven workloads** — scale based on queue depth, not CPU
- **Scale-to-zero** — pay nothing when idle (especially for [GPU inference](/blog/nvidia-nim-support-matrix-models-gpus-profiles-2026/))
- **External metrics** — Kafka lag, SQS depth, database queries
- **Scheduled scaling** — cron-based scale up/down
- **Batch processing** — ScaledJob for queue-draining workers
