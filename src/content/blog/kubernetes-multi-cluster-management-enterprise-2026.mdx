The average enterprise runs 15 Kubernetes clusters. Some run hundreds. Dev, staging, production. Multi-region. Multi-cloud. Edge. GPU clusters for AI. Each one needs consistent policy, security, and observability.

Managing them individually does not scale. Here are the patterns that work.

## Why Multi-Cluster

Single massive clusters seem simpler but fail in practice:

- **Blast radius** — one misconfigured admission webhook takes down everything
- **Compliance boundaries** — PCI, HIPAA, and GDPR data often cannot co-exist in one cluster
- **Team isolation** — namespace isolation is not enough for hostile multi-tenancy
- **Performance** — etcd degrades above ~5,000 nodes; API server latency increases with cluster size
- **GPU specialization** — AI workloads need different node types and scheduling policies

The industry standard: purpose-built clusters with centralized management.

## Pattern 1: GitOps Fleet Management

Use a single Git repository as the source of truth for all clusters:

```
fleet-repo/
├── base/                    # Shared across all clusters
│   ├── namespaces/
│   ├── network-policies/
│   ├── rbac/
│   └── monitoring/
├── overlays/
│   ├── production-eu/       # EU production overrides
│   ├── production-us/       # US production overrides
│   ├── staging/
│   └── gpu-cluster/         # AI workload specifics
└── clusters/
    ├── prod-eu-1.yaml       # ArgoCD ApplicationSet target
    ├── prod-us-1.yaml
    ├── staging-1.yaml
    └── gpu-eu-1.yaml
```

### ArgoCD ApplicationSets

Deploy the same app to all clusters with per-cluster customization:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: platform-base
spec:
  generators:
    - git:
        repoURL: https://github.com/org/fleet-repo
        revision: main
        directories:
          - path: clusters/*
  template:
    metadata:
      name: '{{path.basename}}-platform'
    spec:
      project: platform
      source:
        repoURL: https://github.com/org/fleet-repo
        targetRevision: main
        path: 'overlays/{{path.basename}}'
      destination:
        server: '{{cluster.url}}'
        namespace: platform
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

One commit, all clusters updated. Drift detected and corrected automatically.

## Pattern 2: Cluster API (CAPI)

Manage cluster lifecycle — creation, upgrades, scaling — as Kubernetes resources:

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: prod-eu-2
spec:
  clusterNetwork:
    pods:
      cidrBlocks: ["10.244.0.0/16"]
    services:
      cidrBlocks: ["10.96.0.0/12"]
  controlPlaneRef:
    apiVersion: controlplane.cluster.x-k8s.io/v1beta1
    kind: KubeadmControlPlane
    name: prod-eu-2-control-plane
  infrastructureRef:
    apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
    kind: AzureCluster  # or AWSCluster, GCPCluster, vSphereCluster
    name: prod-eu-2
```

Spin up a new cluster by committing a YAML file. Upgrade Kubernetes versions by changing a field. Delete a cluster by removing the resource.

## Pattern 3: Rancher Fleet

Rancher provides a complete multi-cluster management UI with Fleet for GitOps:

- **Cluster provisioning** — create clusters on any infrastructure from one dashboard
- **Fleet GitOps** — declare desired state, Fleet reconciles across clusters
- **Global RBAC** — manage roles across all clusters from one place
- **Monitoring stack** — Prometheus + Grafana deployed consistently to every cluster

Best for organizations that want a management UI without building everything from scratch.

## Cross-Cluster Networking

Services in one cluster calling services in another:

| Solution | Type | Complexity | Best For |
|----------|------|-----------|----------|
| **Submariner** | L3 tunnel | Medium | Hybrid cloud |
| **Cilium ClusterMesh** | eBPF | Medium | Same cloud provider |
| **Istio Multi-Cluster** | Service mesh | High | Full observability |
| **Skupper** | L7 virtual application network | Low | Simple cross-cluster services |
| **DNS + Ingress** | L7 | Low | Independent services |

## Disaster Recovery

Multi-cluster enables active-active and active-passive DR:

```
Primary (EU)              Secondary (US)
┌──────────────┐         ┌──────────────┐
│ prod-eu-1    │────────▶│ prod-us-1    │
│ Active       │ Velero  │ Standby      │
│ Traffic: 100%│ backups │ Traffic: 0%  │
└──────────────┘         └──────────────┘
         │                       │
    Global LB (Cloudflare/Route53)
    Failover: DNS-based or anycast
```

**Velero** for backup and restore. **External-DNS** for automatic DNS failover. **ArgoCD** ensures the standby cluster is always deployment-ready.
