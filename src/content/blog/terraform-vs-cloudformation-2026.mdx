Terraform is multi-cloud and uses HCL. CloudFormation is AWS-only and uses JSON/YAML. If you are 100% AWS, CloudFormation is free and deeply integrated. If you use multiple clouds (or might), Terraform avoids lock-in.

## Feature comparison

| Feature | Terraform | CloudFormation |
|---|---|---|
| **Provider** | Multi-cloud (4,000+ providers) | AWS only |
| **Language** | HCL | JSON or YAML |
| **State** | State file (you manage) | AWS-managed (automatic) |
| **Drift detection** | `terraform plan` (manual) | Built-in (automatic) |
| **Rollback** | Manual (no auto-rollback) | Automatic on failure |
| **Cost** | Free (CLI), Terraform Cloud (paid) | Free |
| **Preview** | `terraform plan` | Change sets |
| **Modules/reuse** | Modules (Terraform Registry) | Nested stacks, modules |
| **CDK support** | CDKTF (TypeScript, Python, etc.) | AWS CDK (TypeScript, Python, etc.) |
| **Import existing** | `terraform import` | Resource import |
| **License** | BSL 1.1 | Proprietary (AWS) |

## Language comparison

### Terraform (HCL)

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  tags = {
    Name = "production"
  }
}

resource "aws_subnet" "public" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet(aws_vpc.main.cidr_block, 8, count.index)
  availability_zone = data.aws_availability_zones.available.names[count.index]
}

resource "aws_instance" "web" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = var.instance_type
  subnet_id     = aws_subnet.public[0].id
}
```

### CloudFormation (YAML)

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Resources:
  VPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16
      Tags:
        - Key: Name
          Value: production

  PublicSubnet1:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: 10.0.0.0/24
      AvailabilityZone: !Select [0, !GetAZs '']

  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !Ref AmiId
      InstanceType: !Ref InstanceType
      SubnetId: !Ref PublicSubnet1
```

HCL is more concise for loops and conditionals. CloudFormation YAML uses intrinsic functions (`!Ref`, `!Select`, `!GetAZs`) that are harder to read.

## State management

### Terraform

```hcl
# You manage the state file
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "production/terraform.tfstate"
    region         = "eu-west-1"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}
```

You create the S3 bucket, DynamoDB lock table, and manage state file access. State corruption = potential disaster.

### CloudFormation

No state management needed. AWS tracks all resource state internally. You deploy the template, AWS handles everything.

## Rollback behavior

| Scenario | Terraform | CloudFormation |
|---|---|---|
| **Resource creation fails** | Partial state (manual cleanup) | Auto-rollback (all or nothing) |
| **Update fails mid-way** | Partial update (manual fix) | Auto-rollback to previous |
| **Drift detected** | `terraform plan` shows diff | CloudFormation drift detection |

CloudFormation's atomic rollback is a significant safety advantage for production environments.

## Decision guide

### Choose Terraform when:
- **Multi-cloud** — AWS + Azure + GCP (or planning to expand)
- **Non-AWS resources** — Kubernetes, GitHub, Datadog, PagerDuty, DNS
- **Team skill** — your team already knows HCL
- **[OpenTofu](/blog/terraform-vs-opentofu-2026/)** — want an open-source alternative
- You need **advanced logic** (for_each, dynamic blocks, complex modules)

### Choose CloudFormation when:
- **100% AWS** with no multi-cloud plans
- **Zero state management** — AWS handles everything
- **Automatic rollback** on failure is critical
- **AWS integration** — Service Catalog, Control Tower, Organizations
- **Free** — no licensing cost, no state backend to manage
- **AWS CDK** — write infrastructure in TypeScript/Python, compiles to CloudFormation
