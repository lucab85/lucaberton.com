This is a practical comparison based on real production use, not vendor marketing.

## Quick Comparison

| Feature | Terraform | OpenTofu |
|---------|-----------|----------|
| License | BSL 1.1 (source-available) | MPL 2.0 (open source) |
| Governance | HashiCorp (IBM) | Linux Foundation |
| Provider compatibility | Native | Full compatibility |
| State encryption | Terraform Cloud | Built-in (client-side) |
| Registry | registry.terraform.io | registry.opentofu.org + TF registry |
| Early variable evaluation | No | Yes (`-var` in backend config) |
| `removed` block | Added later | Added first |

## When to Use Terraform

- **Terraform Cloud/Enterprise**: If you use HCP Terraform for state management, policy-as-code, and team workflows
- **Enterprise support**: IBM/HashiCorp provides commercial support and SLAs
- **Stability**: Terraform has 10+ years of production track record
- **Existing workflows**: If your CI/CD, modules, and team processes are built around Terraform, switching has a cost

## When to Use OpenTofu

- **Open-source commitment**: If your organization requires OSI-approved licenses
- **Linux Foundation governance**: Community-driven development with transparent decision-making
- **Client-side state encryption**: Built-in encryption at rest for state files without a SaaS dependency
- **Innovation speed**: OpenTofu ships features like early variable evaluation faster

## Migration

OpenTofu is a drop-in replacement for Terraform 1.5.x:

```bash
# Switch from Terraform to OpenTofu
# 1. Install OpenTofu
brew install opentofu

# 2. Initialize (uses existing state)
tofu init

# 3. Plan (same syntax)
tofu plan

# 4. Apply
tofu apply
```

Existing `.tf` files, state files, and providers work without modification.

## My Recommendation

For **new projects** where open-source licensing matters, choose **OpenTofu**. For **existing Terraform deployments** with Terraform Cloud, the switching cost may not be justified.
