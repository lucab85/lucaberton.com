Terraform uses HCL (a domain-specific language). Pulumi uses general-purpose languages (TypeScript, Python, Go, Java, C#). This is the core difference — everything else follows from it.

## Language comparison

### Terraform (HCL)

```hcl
resource "aws_s3_bucket" "data" {
  bucket = "my-data-bucket"
}

resource "aws_s3_bucket_versioning" "data" {
  bucket = aws_s3_bucket.data.id
  versioning_configuration {
    status = "Enabled"
  }
}

# Conditional logic is limited
resource "aws_s3_bucket" "logs" {
  count  = var.enable_logging ? 1 : 0
  bucket = "my-logs-bucket"
}

# Loops use for_each
resource "aws_iam_user" "team" {
  for_each = toset(var.team_members)
  name     = each.value
}
```

### Pulumi (TypeScript)

```typescript
import * as aws from "@pulumi/aws";

const dataBucket = new aws.s3.Bucket("data", {
  versioning: { enabled: true },
});

// Full programming language — conditionals, loops, functions
if (config.requireBoolean("enableLogging")) {
  const logsBucket = new aws.s3.Bucket("logs");
}

// Native loops
const users = ["alice", "bob", "carol"].map(name =>
  new aws.iam.User(name, { name })
);

// Reusable functions
function createVpc(name: string, cidr: string) {
  const vpc = new aws.ec2.Vpc(name, { cidrBlock: cidr });
  const subnet = new aws.ec2.Subnet(`${name}-subnet`, {
    vpcId: vpc.id,
    cidrBlock: cidr.replace("/16", "/24"),
  });
  return { vpc, subnet };
}
```

## Feature comparison

| Feature | Terraform | Pulumi |
|---------|-----------|--------|
| **Language** | HCL (DSL) | TypeScript, Python, Go, Java, C#, YAML |
| **Type safety** | Limited (HCL types) | Full (compiler checks) |
| **IDE support** | Basic (HCL extension) | Full (IntelliSense, refactoring) |
| **Testing** | `terraform test` (basic) | Standard test frameworks (Jest, pytest, Go test) |
| **Loops/conditionals** | `count`, `for_each` (limited) | Native language constructs |
| **State** | State file (S3, TFC) | Pulumi Cloud, S3, local |
| **Providers** | 4,000+ | ~150 native + all Terraform providers via bridge |
| **Plan/preview** | `terraform plan` | `pulumi preview` |
| **Import** | `terraform import` | `pulumi import` |
| **Secrets** | Variables (plaintext in state) | Encrypted in state by default |
| **Policy** | Sentinel (TFC), OPA | CrossGuard (built-in) |
| **Component reuse** | Modules | Classes/functions/packages |
| **Package management** | Terraform Registry | npm, PyPI, Go modules |
| **License** | BSL 1.1 | Apache 2.0 |

## Testing

### Terraform

```hcl
# terraform test (basic assertions)
run "verify_bucket" {
  command = plan
  assert {
    condition     = aws_s3_bucket.data.bucket == "my-data-bucket"
    error_message = "Bucket name mismatch"
  }
}
```

### Pulumi (TypeScript with Jest)

```typescript
import * as pulumi from "@pulumi/pulumi/runtime";
import { describe, it, expect } from "vitest";

describe("Infrastructure", () => {
  it("creates a versioned S3 bucket", async () => {
    const bucket = await getResource("aws:s3:Bucket", "data");
    expect(bucket.versioning.enabled).toBe(true);
  });

  it("encrypts all buckets", async () => {
    const buckets = await getAllResources("aws:s3:Bucket");
    for (const b of buckets) {
      expect(b.serverSideEncryptionConfiguration).toBeDefined();
    }
  });
});
```

Pulumi testing is significantly more powerful — you can use any test framework, mock resources, and test complex logic.

## Secrets handling

Terraform stores secrets in plaintext in state files. You rely on encryption at the state backend level (S3 SSE, Terraform Cloud).

Pulumi encrypts secrets in state by default:

```typescript
const password = new pulumi.Config().requireSecret("dbPassword");
// password is encrypted in state — never stored in plaintext
```

## Decision guide

### Choose Terraform when:

- Your team is **infrastructure-focused** (not developers)
- You want the **largest provider ecosystem** (4,000+ native providers)
- **HCL simplicity** is a feature — it constrains what infrastructure code can do
- You use **[Terraform Cloud](/blog/terraform-vs-opentofu-2026/)** (Sentinel, cost estimation, VCS workflows)
- Existing codebases are in HCL — migration cost is high

### Choose Pulumi when:

- Your team is **developer-heavy** and already knows TypeScript/Python/Go
- You need **complex logic** — loops, conditionals, API calls during provisioning
- **Type safety and IDE support** are important (catch errors before plan)
- You want **real testing** with standard test frameworks
- **Secrets encryption** in state is a requirement
- You want to share infrastructure code via **npm/PyPI packages**
